# Agent方案 #
将 FlowCut 从"配置式批量剪辑工具"升级为"自动剪辑 Agent"，本质上是将"人填表单 → 机器执行"转变为"人给目标 → 机器规划并执行"。
这不是推翻重写，而是在现有后端能力（素材库、FFmpeg 流水线、TTS）之上，加一层**大模型大脑**和**工具调用**框架。
以下是完整演进路径和架构设计：

---
## 一、核心概念转变
| 维度 | 当前 FlowCut（脚本式） | FlowCut Agent（智能体式） |
|------|----------------------|--------------------------|
| 交互方式 | 5步表单向导（选主题→选素材→配参数） | 自然语言对话（"帮我剪5条奶茶广告"） |
| 素材选择 | 人眼浏览，手动勾选 | Agent 根据语义理解自动检索和挑选 |
| 文案生成 | 预设文案库循环取用 | Agent 根据素材内容和受众即时生成 |
| 参数配置 | 人手动调下拉框 | Agent 根据平台调性自动决策（如抖音偏好快节奏） |
| 质量把控 | 剪完才看到，不满意重新配 | Agent 剪完自审，不满意自动换素材重剪 |
---
## 二、Agent 架构设计
采用经典的 **ReAct（Reasoning + Acting）** 架构：
```
┌─────────────────────────────────────────────────────────┐
│                      用户输入                            │
│  "帮我剪3条火锅广告，要冬天聚餐氛围，快节奏，发抖音"      │
└─────────────────────┬───────────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────────┐
│                   LLM 大脑 (规划器)                      │
│                                                         │
│  1. 意图解析：主题=火锅，氛围=冬天聚餐，节奏=快，平台=抖音  │
│  2. 策略推导：                                            │
│     - 抖音快节奏 → 单条10s，转场0.3s，动感配乐             │
│     - 冬天聚餐 → 优先选沸腾、围坐、热气素材                 │
│     - 需要先搜素材，再挑选，再生成文案，最后渲染            │
│  3. 生成 Tool Call Plan                                   │
└─────────────────────┬───────────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────────┐
│                 Tool 执行层 (现有后端封装)                 │
│                                                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │搜索素材   │ │分析素材   │ │生成文案   │ │配置渲染   │  │
│  │Tool      │ │Tool      │ │Tool      │ │Tool      │  │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐               │
│  │生成配音   │ │执行渲染   │ │质检评估   │               │
│  │Tool      │ │Tool      │ │Tool      │               │
│  └──────────┘ └──────────┘ └──────────┘               │
└─────────────────────┬───────────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────────┐
│                 记忆层 (Memory)                          │
│  短期记忆：当前对话上下文、当前任务状态                    │
│  长期记忆：品牌规范(VDB)、历史高质量产出、用户偏好        │
└─────────────────────────────────────────────────────────┘
```

## 三、关键 Tool 设计（对接现有后端）
将现有的 FastAPI 接口封装为 LLM 可调用的 Tool（Function Calling）：
### 3.1 素材相关
```json
{
  "name": "search_materials",
  "description": "从素材库中搜索视频素材。可根据关键词、场景语义、时长筛选。",
  "parameters": {
    "query": "沸腾的火锅红油",
    "tags": ["火锅", "特写"],
    "min_duration": 3,
    "max_duration": 10,
    "platform": "抖音",
    "limit": 10
  }
}
```
```json
{
  "name": "analyze_material",
  "description": "分析视频素材的内容，返回场景描述、色调、运动强度（用于判断节奏匹配度）。",
  "parameters": {
    "material_id": 123
  }
}
// 返回：{ "scene": "几个人围着火锅夹肉，有热气", "tone": "暖黄色", "motion": "高" }
```
### 3.2 创意相关
```json
{
  "name": "generate_ad_copy",
  "description": "根据素材内容和营销目标生成广告文案，返回分句结果及每句建议时长。",
  "parameters": {
    "theme": "火锅",
    "scene_description": "沸腾的红油、朋友们举杯、热气腾腾",
    "tone": "热情、接地气",
    "platform": "抖音",
    "count": 3
  }
}
// 返回：[{ "text": "这顿火锅，暖了整个冬天", "suggested_duration": 3.5 }, ...]
```
### 3.3 渲染相关
```json
{
  "name": "render_video",
  "description": "提交视频渲染任务。传入挑选的素材ID列表、文案时间线、技术参数。",
  "parameters": {
    "clips": [
      { "material_id": 123, "start_sec": 1.0, "end_sec": 4.5 },
      { "material_id": 456, "start_sec": 0.0, "end_sec": 3.0 }
    ],
    "subtitle_segments": [
      { "text": "这顿火锅", "start": 0.0, "end": 2.0 },
      { "text": "暖了整个冬天", "start": 2.0, "end": 5.0 }
    ],
    "config": {
      "transition": { "type": "滑动", "duration": "0.3s" },
      "music": { "style": "动感", "volume": 25 },
      "output": { "ratio": "9:16", "resolution": "1080p" }
    }
  }
}
```
### 3.4 质检相关（闭环关键）
```json
{
  "name": "evaluate_video",
  "description": "对产出的视频进行质量评估，检查画面模糊、字幕遮挡、节奏等。",
  "parameters": {
    "task_output_id": 789
  }
}
// 返回：{ "score": 85, "issues": ["第3秒画面较暗"], "pass": true }
```

## 四、最难攻克的技术点
### 4.1 视频理解（Agent 的"眼睛"）
LLM 看不懂视频，这是最大瓶颈。两种解决路径：
**方案 A：入库时预标注（推荐，低成本）**
- 素材上传时，抽取 3-5 个关键帧
- 调用视觉模型（如 Qwen-VL / GPT-4o-mini）生成描述
- 将描述存入向量数据库
- Agent 搜索时搜文本描述即可，不需要实时看视频
```python
# 素材入库时的预标注流程
frames = extract_keyframes(video_path)  # 抽取关键帧
descriptions = []
for frame in frames:
    desc = vl_model.describe(frame)  # "一个玻璃杯里装满棕色奶茶，有冰块"
    descriptions.append(desc)
full_desc = " | ".join(descriptions)
embedding = embed_model.encode(full_desc)
vector_db.insert(material_id, embedding, full_desc)
```
**方案 B：实时多模态调用（高精度，高成本）**
- Agent 在挑选素材时，将关键帧发给 GPT-4V 判断是否匹配当前需求
- 适合对质量要求极高的场景
### 4.2 Agent 自主决策的确定性控制
Agent 不能"自由发挥"，必须有边界：
| 控制点 | 策略 |
|--------|------|
| 素材版权 | Tool 层面限制，只能从素材库选，不能自己编造素材 |
| 技术参数边界 | 系统提示词中写死规则，如"抖音信息流必须 9:16，时长不超过 30s" |
| 文案安全 | 生成文案后过一遍敏感词过滤 Tool，不通过则重新生成 |
| 成本控制 | 限制 ReAct 循环次数（如最多 15 轮），超次强制输出当前最佳结果 |

## 六、分阶段演进路线
不要一步到位，建议分三期走：
### Phase 1：Copilot 模式（2 周）— 最小改动
**保留现有 UI 不变，加一个 AI 助手侧边栏。**
- 用户在素材库浏览时，AI 根据当前筛选条件推荐素材
- 用户在工作流 Step3 配参数时，AI 根据"主题+平台"一键推荐最佳参数组合
- 用户输入一句话（如"我要快节奏的"），AI 自动填充表单字段
- **技术改动**：纯前端加一个对话组件，调 LLM API，LLM 输出结构化 JSON，前端用 JS 填充表单。后端零改动。
### Phase 2：Tool Agent 模式（3-4 周）— 核心重构
**去掉表单，改为对话式输入。**
- 后端引入 LangChain 或自研轻量 Agent 框架
- 将现有 FastAPI 接口封装为 Tool
- 实现 ReAct 循环：用户输入 → LLM 规划 → 调用 Tool → 返回结果
- 加入素材预标注管线（入库时抽取关键帧 + VLM 描述 + 向量化）
- **技术栈新增**：LangChain / LlamaIndex、多模态模型 API、向量数据库（Milvus / pgvector）
### Phase 3：自主闭环模式（4+ 周）— 高级能力
- **质检闭环**：剪完后 Agent 自己看一遍（调 VLM 看产出封面/关键帧），不满意自动换素材重剪
- **品牌记忆**：向量数据库存储品牌规范（"XX品牌字幕必须在底部居中，不能用黄色"），Agent 自动遵循
- **多轮优化**：用户说"感觉不够高级"，Agent 理解"高级"→ 换舒缓配乐 + 慢转场 + 调低饱和度 → 重新渲染
- **批量 A/B**：Agent 自动生成同一主题的不同风格变体（3条快节奏 + 3条抒情 + 3条搞笑），供投放测试

## 七、技术栈补充
在现有技术栈基础上，Agent 阶段需要新增：
| 能力 | 技术选型 | 说明 |
|------|----------|------|
| LLM 大脑 | Qwen-Max / GPT-4o | 需要强工具调用能力 |
| Agent 框架 | LangChain / 自研 | 管理 Tool 调用链和上下文 |
| 视频理解 | Qwen-VL / GPT-4o-mini | 素材入库时描述关键帧 |
| 向量数据库 | PgSQL + pgvector | 存储素材语义向量，不用新增组件 |
| 文本向量 | text-embedding-3-small | 素材描述和品牌规范的向量化 |
| Prompt 管理 | 独立 YAML / 数据库 | 系统提示词、平台规则、品牌规范的结构化管理 |

## 八、成本估算
Agent 模式的主要增量成本在 LLM 调用和视频理解：
| 环节 | 单次任务消耗 | 单价估算 | 单任务成本 |
|------|-------------|----------|-----------|
| 意图理解 + 规划 | ~2000 tokens | ¥0.01/千token | ¥0.02 |
| 素材预标注（入库时，非单次） | ~500 tokens/帧 × 5帧 | ¥0.005/千token | ¥0.0125（摊销） |
| 文案生成 | ~500 tokens | ¥0.01/千token | ¥0.005 |
| 质检评估（可选） | ~1000 tokens | ¥0.01/千token | ¥0.01 |
| **单任务总计** | | | **¥0.03 - 0.05** |
对比人工剪辑（20分钟/条 × 时薪50元 ≈ ¥17/条），Agent 增量成本可忽略不计。

## 总结建议
**如果你现在要启动 Agent 化，第一件事不是写代码，而是：**
1. **给素材打标签**：跑一个脚本，把现有素材全部抽关键帧送 VLM 生成描述，存入 pgvector。这是 Agent 能"看懂"素材的前提，没有这个，后面全是空谈。
2. **做 Phase 1 Copilot**：花 2 周在现有 UI 上加个对话框，验证"LLM 能不能理解剪辑需求并正确填表"。这一步能过滤掉 80% 的伪需求。
3. **封装 Tool API**：把现有的 `/api/materials`、`/api/tasks` 改造为严格的 JSON Schema 描述，能被 LLM Function Calling 直接调用。
